# AI 文献总结与问答助手（基于 RAG）— 技术需求文档（TRD）

**文档版本**：1.1  
**文档类型**：技术需求与实现约束（MVP）  
**关联文档**：[产品需求文档（PRD）](./need-product.md)

---

## 1. 目标与范围

本文档规定 MVP 的**系统架构、数据与接口、算法与工程约束、非功能需求**，用于研发实现与测试验收。产品行为与验收口径以 PRD 为准；二者冲突时以 PRD 的用户承诺为准，TRD 补充可落地的技术方案。

---

## 2. 系统架构

### 2.1 逻辑模块

| 模块 | 技术职责 |
|------|----------|
| **文档处理** | PDF 解析、文本清洗、分段（chunking）、页码/段落等元数据抽取 |
| **向量存储** | 文本 embedding、向量索引构建与持久化；MVP 推荐 **FAISS**（或等价向量检索方案） |
| **检索** | 相似度检索、top-k chunk 返回、可选分数阈值过滤 |
| **大模型调用** | 总结与问答的 Prompt 编排、流式/非流式接口、超时与重试 |
| **API 服务层** | 建议 **FastAPI**：上传、索引构建、总结、问答、健康检查 |

### 2.2 部署形态（MVP）

- 单体或「API + 可选轻量前端」均可；须明确 **PDF / 索引 / 日志** 的存储路径与清理策略。  
- 密钥与模型 API Key **禁止**写入仓库；使用环境变量或密钥管理。  

---

## 3. 核心流程（实现级）

### 3.1 文献总结流水线

```
上传 PDF → 解析为纯文本（带页码映射）→ 清洗 →（可选）chunk 缓存
    → 全文或压缩上下文送入 LLM → 结构化 JSON/分段文本 → 持久化摘要（可选）
```

### 3.2 RAG 问答流水线

```
用户问题 → 问题 embedding → FAISS top-k 检索 → 按策略拼接上下文
    → LLM 生成（结构化 + 引用字段）→ 响应（答案 + citations）
```

### 3.3 索引构建时机

- 上传成功后**同步或异步**构建向量索引；异步时须暴露**任务状态**或**就绪查询**，避免用户在索引未完成时得到空检索结果且无提示。

## 4. 数据与元数据

### 4.1 Chunk 策略

- 切分方式：**按段落优先**；过长段落再按固定窗口切分。  
- 目标规模：约 **500～1000 tokens** 每块（或与所选 embedding 模型的最大序列对齐，取较小安全值）；若以字符近似，需在实现中写明换算规则。  
- 尽量避免在句子中间截断；不可避免时保留重叠窗口（overlap）以减少语义断裂（推荐 overlap 为块大小的 10%～20%，可配置）。  

### 4.2 Chunk 元数据（最低集）

每条 chunk 至少包含：

| 字段 | 说明 |
|------|------|
| `chunk_id` | 全局或文档内唯一 ID |
| `document_id` | 文献唯一标识 |
| `text` | 块正文 |
| `page_start` / `page_end` | 若解析器支持则必填；不支持则填 `null` 并在 API 中显式返回 |
| `paragraph_index` | 可选，段落序号 |
| `source_offset` | 可选，字符偏移，便于未来高亮 |

页码与段落无法从 PDF 可靠获取时，须在响应中 `location_note` 类字段说明，与 PRD「无则说明」一致。

### 4.3 摘要缓存（可选但推荐）

- 对同一 `document_id` 的结构化摘要可缓存，供模型失败时 **降级返回**（与 PRD 6.4 一致）。

## 5. API 需求（MVP 级契约）

以下路径为示例，实现时可调整但须在 README/OpenAPI 中固定文档。

| 能力 | 方法 / 路径（示例） | 请求要点 | 响应要点 |
|------|---------------------|----------|----------|
| 上传 PDF | `POST /documents` | `multipart/form-data`，文件字段名约定 | `document_id`、处理状态 |
| 索引状态 | `GET /documents/{id}/status` | — | `pending` / `ready` / `failed`、错误码 |
| 文献总结 | `POST /documents/{id}/summarize` | 可选语言/模板版本 | 结构化字段 + 可选 `cached` 标记 |
| RAG 问答 | `POST /documents/{id}/query` | `question`、可选 `session_id`、`top_k` | `answer`、`citations[]`（含 `chunk_id`、`excerpt`、`page_*`） |

### 5.1 错误与 HTTP 语义

- 解析失败：`4xx` + 明确 `error_code`（如 `PDF_PARSE_FAILED`、`EMPTY_TEXT`）。  
- 模型超时：可 `503` 或 `200` + 业务字段 `degraded: true` + 缓存摘要；**禁止**用虚构正文填充。  
- 检索无结果：业务成功 `200`，`answer` 为「无法从文献中找到依据」类结构化内容，`citations` 为空数组。  

### 5.2 鉴权与限流（建议）

- MVP 可单用户无鉴权；若暴露公网，须至少 **API Key** 或反向代理限流。  

---

## 6. 检索与生成约束

### 6.1 检索参数

- 默认 `top_k`：建议 **4～8**（可配置）。  
- **相似度阈值**：低于阈值视为「无相关 chunk」，走 PRD 6.3 话术；阈值需在配置文件中可调并记录日志。  

### 6.2 Prompt 工程（强制）

- 系统提示中须包含：**仅基于提供的 Context 回答**；Context 未提及则明确说不知道或未记载。  
- 输出格式建议为 **JSON Schema** 或带固定分隔符的结构，便于解析 `answer` 与 `citations`。  
- 要求模型在回答中引用 `chunk_id` 或与 chunk 列表索引对应，便于前端展示。  

### 6.3 幻觉控制（工程侧）

- **禁止**在无检索上下文时调用「纯聊天」路径回答文献问题（除非 PRD 定义的澄清类交互单独建模）。  
- 对「与文献无关」的判定：可采用轻量分类器或 LLM 路由；若成本敏感，可 MVP 仅用「检索分数 + 拒绝模板」近似。  

---

## 7. 非功能需求

### 7.1 性能

- 在单篇典型论文（如 ≤ 30 页、文本可提取）下，**总结与单次问答** P95 目标 **5 秒以内**（不含用户上传带宽）；超出时记录耗时分解（解析 / embedding / 检索 / LLM）。  
- 首次全量 embedding 可异步；须符合第 3.3 节状态接口。  

### 7.2 可观测性

- 结构化日志：请求 ID、`document_id`、各阶段耗时、模型错误码、检索 top-1 分数。  
- 不在日志中输出完整用户 PDF 正文；可截断或仅记录 hash。  

### 7.3 依赖选型（MVP 建议）

| 类别 | 建议 |
|------|------|
| Web 框架 | FastAPI |
| 向量库 | FAISS（内存或 mmap 持久化） |
| PDF | pdf.js / PyMuPDF / pdfminer 等择一并评估中文与双栏论文 |
| Embedding / LLM | 通过统一抽象层接入，便于切换供应商 |

---

## 8. 测试与验收（技术侧）

- **单元测试**：chunk 边界、空文本、损坏 PDF。  
- **集成测试**：上传 → ready → summarize → query 全链路。  
- **RAG 回归**：固定 PDF + 固定问题，断言响应中含至少一条 citation 且 excerpt 为原文子串（允许规范化空白）。  

---

## 9. 安全与合规（基线）

- 用户上传文件大小与页数上限；拒绝可执行附件伪装。  
- 临时文件定期清理。  
- 若未来支持多用户，须隔离 `document_id` 与索引文件权限。
