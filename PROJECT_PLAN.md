# AI 文献总结助手项目方案

## 一句话定义

一个帮助用户快速判断论文价值、理解核心创新，并转化为后续行动的 AI 研究阅读助手。

## 为什么要做

现有很多“论文总结工具”只能提供泛泛摘要，但真实用户更关心的是：

- 这篇论文解决了什么实际问题
- 它和已有方法相比到底新在哪里
- 实验结果是否可信
- 我有没有必要继续深读、复现或用于产品设计

所以这个项目的目标不是“生成摘要”，而是“帮助用户做研究决策”。

## 目标用户

### 1. 学生 / 入门者

痛点：

- 论文读不动，术语和公式门槛高
- 不知道该按什么顺序读
- 读完后也难以复述

核心价值：

- 降低阅读门槛
- 给出推荐阅读路径
- 帮助建立基础概念框架

### 2. 算法工程师 / 研究员

痛点：

- 每天需要快速筛选大量论文
- 想抓创新点和实验设计，但人工阅读耗时高
- 需要判断是否值得复现

核心价值：

- 快速提取方法创新
- 提醒实验风险与不足
- 帮助做论文优先级排序

### 3. AI 产品经理

痛点：

- 能读懂技术，但不容易快速转成产品洞察
- 想知道论文对用户体验和业务能力的影响
- 需要将论文内容讲给非技术同事听

核心价值：

- 把论文翻译成产品机会
- 输出汇报式表达
- 支持后续 Demo 和实验设计

## MVP 范围

第一版先解决“单篇论文快速理解”问题，包含：

1. 输入论文标题、摘要或核心内容
2. 选择用户角色
3. 选择阅读目标
4. 输出结构化结果

输出内容建议固定为 5 个模块：

1. 一句话总结
2. 核心创新点
3. 实验与风险提示
4. 落地启发
5. 汇报提纲

## 关键功能设计

### 功能 1：角色化总结

同一篇论文，对不同用户输出不同结果。

- 学生：强调概念解释和阅读顺序
- 研究员：强调方法细节、实验结论和复现价值
- AI PM：强调产品机会、业务影响和用户价值

### 功能 2：目标导向输出

用户不是只想“看摘要”，而是带着目标来：

- 值不值得深入阅读
- 能不能用于组会或面试汇报
- 能不能转化为产品灵感

所以摘要应该根据目标变化表达方式。

### 功能 3：风险意识

相比很多只讲亮点的摘要器，这个产品会明确提示：

- 论文结论的适用边界
- benchmark 与真实场景的差异
- 复现或落地的潜在成本

这会更像一个“决策助手”，而不是“信息搬运工具”。

## 产品差异化

和普通 AI 总结工具相比，这个项目的差异在于：

- 从“摘要”升级到“决策支持”
- 从单一输出升级到 persona 输出
- 从信息整理升级到行动建议
- 从工具思路升级到产品思路

## 核心指标

如果后续继续迭代，可以关注这些指标：

### 北极星指标

- 用户完成一次论文判断所需时间下降

### 过程指标

- 论文阅读完成率
- 用户点击“继续深读”的比例
- 用户保存总结结果的比例
- 用户生成汇报提纲的比例

### 质量指标

- 用户对摘要准确性的主观评分
- 用户对“是否值得读”判断的认可度
- 不同 persona 输出的满意度差异

## 后续版本规划

### V1

- 单篇论文输入
- 结构化总结
- Persona 切换
- 汇报提纲输出

### V2

- PDF 上传与自动解析
- 支持引用、图表和实验章节抽取
- 支持真实 LLM API 动态生成

### V3

- 多篇论文对比
- 自动生成研究脉络
- 收藏、标签、知识库沉淀

### V4

- 与 Notion / 飞书 / Obsidian 联动
- 支持团队协作评注
- 根据用户历史行为做个性化推荐

## 面试表达建议

你可以这样讲这个项目：

1. 我没有把问题定义成“做一个论文摘要器”，而是定义成“帮助用户更快做研究决策”
2. 我先拆分了目标用户，因为不同用户需要完全不同的信息密度和表达方式
3. 第一版我优先验证的是交互闭环和输出结构，而不是一开始就堆技术
4. 如果继续做，我会接真实模型、做用户访谈，并通过阅读时长、保存率和满意度验证产品价值

## 技术落地建议

如果继续开发成更完整版本，推荐技术路线：

- 前端：Next.js 或 React
- 样式：Tailwind 或原生 CSS 设计系统
- 文档解析：PDF 提取 + 分块
- 模型层：OpenAI 或兼容 API
- 存储：Supabase / Firebase
- 评估：人工标注 + Prompt 版本对比

## 当前仓库价值

这个仓库已经具备两个直接用途：

1. 作为面试时可直接打开演示的作品
2. 作为后续接真实模型和后端能力的前端原型
